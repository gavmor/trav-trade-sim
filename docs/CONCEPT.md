# Traveller Trade Simulator — Concept

## Overview

Traveller Trade Simulator (TTS) is a web-based speculative trade dashboard for tabletop roleplaying campaigns set in the Traveller universe. It provides a shared, persistent market economy that the Referee manages and players interact with in real time, replacing the paper-and-dice mechanics of Classic Traveller Book 7 and Traveller 5th Edition with a tracked, historically-charted digital equivalent.

## Problem Statement

Speculative trade in Traveller is one of the game's most engaging subsystems, but it is traditionally managed with dice rolls, lookup tables, and handwritten ledgers. This creates three problems:

1. **No market history.** Players cannot see price trends, identify seasonal patterns, or plan routes using historical data. Every decision is made blind.
2. **No shared state.** In multi-player campaigns, each character may be tracking their own holdings separately with no reconciliation between sessions.
3. **High mechanical overhead.** Computing purchase price, actual value rolls, TL adjustments, broker fees, and quantity availability for 36 trade goods per world is tedious enough that many groups skip it entirely.

## Goals

| Goal | Description |
|------|-------------|
| **G-1** | Eliminate trade calculation overhead so players spend session time on decisions, not arithmetic |
| **G-2** | Provide persistent, shared market state that all players in a campaign see identically |
| **G-3** | Enable route analysis — show projected profit for every reachable world before committing to a jump |
| **G-4** | Produce a living market that changes over time via M.U.L.E.-style random events and Referee-created events |
| **G-5** | Support multiple Traveller editions (Classic CT7, Traveller 5th Edition) in a single tool |
| **G-6** | Require no server infrastructure beyond a Cloudflare free-tier account (Workers + D1) |
| **G-7** | Run entirely in a web browser with no installation |

## Non-Goals

- Full character creation or advancement tracking
- Ship combat or physical maintenance systems (annual maintenance rolls, misjump risk from skipped upkeep)
- Real-money transactions or monetisation of any kind
- Mobile-native app (responsive web only)

## Intended Users

**Referee** — One per campaign. Creates the campaign, manages ships and crew, creates events, and advances the in-game clock. Has access to a dedicated management panel.

**Players** — One or more per campaign. Each controls a character assigned to a ship. Can view the market, buy and sell cargo, analyse routes, and review price history.

## Core Concepts

### The Tick

Time advances in discrete units called ticks. One tick equals one jump-week (7 Imperial days). The Referee advances the tick manually, typically once per in-game week. Prices update on every tick; monthly and annual OHLC (open/high/low/close) rollups fire automatically at the appropriate intervals.

### Deterministic Prices

Market prices are generated by a seeded pseudo-random number generator keyed on `(campaignId, worldHex, goodDie, tick)`. This guarantees that every client produces identical prices for the same inputs without server-side randomness. Prices are stored in the D1 database once generated and never recalculated, ensuring historical integrity.

### The Imperial Calendar

The in-game date is displayed as `DDD-YYYY` (day-of-year and year) per Traveller convention. The campaign's starting year is set at creation and controls the date offset. Classic Traveller campaigns default to year 1105; other milieux can start at any year from 0 to 2500.

### Market Events (M.U.L.E.-style)

When a player first visits a world's market at a new tick, the system probabilistically generates a market event using the same seeded RNG. Events apply a percentage price modifier to one good or all goods at local or subsector scope, lasting for a defined number of ticks. Events are classified as Minor (~4% per-world per-tick), Major (~1.5%), or Crisis (~0.4%).

The Referee can also create events manually — useful for campaign-specific story beats such as trade embargoes, noble patronage, or Imperial interdicts.

### Ships and Crew

Ships are the unit of trade operation. Each ship has a cargo capacity, an operating credit account, and a current location. Characters are assigned to ships as crew with roles (captain, pilot, engineer, etc.). The `can_trade` flag controls who may execute buy/sell transactions; captains receive it automatically. A ship also carries a referee-entered market valuation, may jointly be owned by multiple players in recorded percentage shares, and may carry outstanding debts (mortgages, loans) that a Net Worth report nets against its credits and cargo value.

### Organizations

Players can found an Organization — a generic entity covering corporations, confederations, and trade unions alike, differentiated only by configuration rather than separate systems. An organization has its own treasury, can collect flat-rate dues from and disburse funds to its member ships, and may own a member ship outright (in which case that ship's value flows into its owning players' personal Net Worth via their equity stake in the organization, rather than the ship's own ownership shares). Any player can found one and becomes its first officer; day-to-day management is open to any of the organization's officers, not gated behind the Referee.

### Route Analysis

From any world, the Jump tab shows all reachable worlds within the ship's jump range, with projected best trade goods and estimated profit per route. Selecting a destination commits the ship's location and navigates to the Market tab for that world, representing arrival after a jump.

## Trade Rules Supported

| Code | Source | Description |
|------|--------|-------------|
| `CT7` | Classic Traveller Book 7 (Merchant Prince) | Cost-of-goods table, market price table, actual value table, broker DM and fee |
| `T5`  | Traveller 5th Edition | T5 trade classification list and pricing rules |

Trade rules are chosen at campaign creation and are immutable. Both rule sets use the same calendar, event, and ship systems.

## Market Data Lifecycle

```
Tick advances (Referee)
  └─► Cache invalidated
        └─► Player opens Market tab for a world
              ├─► Check D1 for existing snapshots at this tick
              ├─► If none: maybe generate event → generate 36 snapshots → insert via Worker
              ├─► On first-ever visit: backfill entire current year for chart context
              └─► Load snapshots → display MarketTable + PriceChart
```
